## 二、核心模块详细评测

### 2.1 `mcpos/adapters/library_renamer.py`

#### 📋 功能概述
- **主要功能**：多频道歌库批量重命名工具
- **核心职责**：
  1. 从文件名提取基础标题
  2. 检测并修复问题标题（"New New"、数字重复、以数字结尾等）
  3. 使用 API 或模板库生成新标题
  4. 确保标题唯一性
  5. 生成艺术家名字和版本后缀
  6. 重命名为统一格式：`标题 - 艺术家[版本后缀].mp3`

#### ✅ 优点

1. **架构设计优秀**
   - ✅ 遵循 McPOS 自包含原则（所有逻辑在 `mcpos/` 下）
   - ✅ 边界模块设计（通过 adapter 暴露接口）
   - ✅ 支持频道特定配置（`channels/{channel_id}/config/library_rename.json`）
   - ✅ 函数职责清晰，模块化良好

2. **功能完整**
   - ✅ 多频道支持（RBR、Kat 等）
   - ✅ 自动问题检测（7种问题类型）
   - ✅ API + 模板库双重保障
   - ✅ 标题唯一性保证
   - ✅ 艺术家分配算法（限制每个艺术家最多3首歌）
   - ✅ 版本后缀支持（remix、ft.、Extended Version 等）

3. **用户体验好**
   - ✅ 默认 dry-run 模式（安全）
   - ✅ 进度显示（每50个文件显示一次）
   - ✅ CSV 计划输出（可预览）
   - ✅ 错误处理完善

4. **代码质量**
   - ✅ 类型注解完整
   - ✅ 函数文档清晰
   - ✅ 配置与逻辑分离
   - ✅ 可扩展性强

#### ⚠️ 缺点和改进建议

1. **性能问题**
   - ⚠️ 每个文件都调用 `ffprobe` 获取时长（可能较慢）
   - 💡 **建议**：批量获取时长，或使用缓存

2. **API 调用优化**
   - ⚠️ API 调用是同步的，可能较慢
   - 💡 **建议**：考虑异步调用或批量处理

3. **错误处理**
   - ⚠️ 某些异常可能被静默忽略（如 `get_audio_duration_seconds`）
   - 💡 **建议**：添加更详细的错误日志

4. **配置管理**
   - ⚠️ 默认配置硬编码在代码中（`DEFAULT_CONFIG`）
   - 💡 **建议**：考虑将默认配置移到外部文件

5. **测试覆盖**
   - ⚠️ 未看到测试文件
   - 💡 **建议**：添加单元测试，特别是问题检测和标题生成逻辑

## 六、性能分析

### 6.1 当前性能瓶颈

1. **音频时长获取**
   - 每个文件调用 `ffprobe`（~0.1-0.5秒/文件）
   - 对于 400+ 文件，可能需要 40-200 秒
   - **优化建议**：批量获取或使用缓存

2. **API 调用**
   - 同步调用，每个问题标题需要 ~1-3 秒
   - 如果有 50 个问题标题，需要 50-150 秒
   - **优化建议**：异步调用或批量处理

3. **文件重命名**
   - 文件系统操作，通常很快（<0.01秒/文件）
   - 不是瓶颈

### 6.2 性能优化建议

1. **批量获取时长**
   ```python
   # 建议：一次性获取所有文件的时长
   durations = batch_get_audio_durations(file_paths)
   ```

2. **异步 API 调用**
   ```python
   # 建议：使用 asyncio 并发调用 API
   async def batch_fix_titles(titles):
       tasks = [fix_title_async(title) for title in titles]
       return await asyncio.gather(*tasks)
   ```

3. **缓存机制**
   ```python
   # 建议：缓存已处理的文件信息
   @lru_cache(maxsize=1000)
   def get_audio_duration_cached(file_path):
       ...
   ```
