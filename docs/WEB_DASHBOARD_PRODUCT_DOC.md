## 1. 仪表板现状分析

### 1.1 页面结构

当前仪表板 (`http://127.0.0.1:8080/`) 是一个单页面实时监控界面，采用深色主题（背景色 #1a1a1a），布局为响应式网格系统。

#### 页面区域划分

```
┌─────────────────────────────────────────────────────────────┐
│  🎵 Kat Records - 实时仪表板         最后更新: HH:MM:SS     │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│  │ 📊 概览   │  │ ⏱️ 阶段耗时│ │ 🔄 活跃阶段│                │
│  └──────────┘  └──────────┘  └──────────┘                 │
│                                                              │
│  ┌──────────────────────┐  ┌──────────────────────┐       │
│  │ 📋 期数列表           │  │ 📰 最近事件          │       │
│  │ (可滚动列表)          │  │ (可滚动列表)         │       │
│  └──────────────────────┘  └──────────────────────┘       │
│                                                              │
│  自动刷新: 每10秒 | 最后更新: HH:MM:SS                      │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 当前功能范围

✅ **已实现：**
- 实时数据刷新（10秒间隔）
- 期数状态概览统计
- 期数列表展示
- 事件流监控
- 失败期数恢复功能

⚠️ **部分实现：**
- 阶段耗时显示（数据缺失时显示 `--`）
- 活跃阶段统计

❌ **未实现：**
- 歌库管理
- 图库管理
- 节目详情页
- 数据分析功能
- 设置页面

## 2. 数据元素来源详解

### 2.1 顶部区域

#### "🎵 Kat Records - 实时仪表板"
- **类型:** 静态文本
- **来源:** HTML 硬编码 (`dashboard.html`)
- **逻辑:** 页面标题，不变化

#### "最后更新: HH:MM:SS"
- **类型:** 动态时间戳
- **来源:** 前端 JavaScript `new Date().toLocaleTimeString()`
- **逻辑:** 每次 `refreshDashboard()` 函数执行后更新
- **更新频率:** 10秒（跟随自动刷新）

---

### 2.2 📊 概览面板

**API端点:** `GET /metrics/summary?period=24h`  
**后端实现:** `dashboard_server.py:get_summary()` 或 `metrics_api.py:get_summary()`

#### "总期数"
- **数据来源:** `summary.global_state.total_episodes`
- **后端逻辑:** 
  1. 从 `config/schedule_master.json` 加载数据
  2. 获取 `episodes` 数组长度
  3. 统计所有期数（不论状态）
- **数据文件:** `config/schedule_master.json`
- **更新时机:** 排播表创建或更新时

#### "已完成"
- **数据来源:** `summary.global_state.completed`
- **后端逻辑:**
  1. 从 `config/schedule_master.json` 获取所有 `episodes`
  2. 遍历每个期数，检查 `status` 字段
  3. 状态映射：`"已完成" → "completed"`
  4. 统计 `status === "已完成"` 的数量
- **状态映射表:**
  ```python
  {
      "待制作": "pending",
      "制作中": "remixing",
      "上传中": "uploading",
      "排播完毕待播出": "uploading",
      "已完成": "completed",  # ← 映射目标
      "已跳过": "pending"
  }
  ```
- **前端显示:** 绿色数字（`.stat-value.success`）

#### "失败"
- **数据来源:** `summary.global_state.error`
- **后端逻辑:** 统计 `status === "失败"` 或 `"error"` 的期数
- **前端显示:** 红色数字（`.stat-value.error`）

#### "进行中"
- **数据来源:** `(summary.global_state.remixing || 0) + (summary.global_state.rendering || 0)`
- **前端逻辑:** 前端JavaScript计算两个活跃阶段的总和
- **后端提供:** 
  - `remixing`: `status === "制作中"` 的数量
  - `rendering`: `status === "渲染中"` 的数量
- **前端显示:** 黄色数字（`.stat-value.warning`）

#### "成功率"
- **数据来源:** 前端计算
- **计算公式:** `(completed / (completed + error)) * 100%`
- **前端逻辑:**
  ```javascript
  const total = gs.completed + gs.error;
  const successRate = total > 0 
      ? ((gs.completed / total) * 100).toFixed(1) + '%'
      : '0%';
  ```
- **前端显示:** 绿色百分比（`.stat-value.success`）

---

### 2.3 ⏱️ 阶段耗时面板

**API端点:** `GET /metrics/summary?period=24h`  
**数据来源:** `summary.stages` 对象

#### 当前状态
- **显示内容:** `--`（占位符）
- **原因:** 
  1. `metrics.json` 中缺少阶段耗时数据
  2. 或 `metrics_manager.get_summary()` 未返回 `stages` 数据

#### 预期数据结构
```json
{
  "stages": {
    "remixing": {
      "count": 4,
      "completed": 4,
      "failed": 0,
      "total_duration": 1200.5,
      "avg_duration": 300.125
    },
    "rendering": {
      "count": 3,
      "avg_duration": 450.0
    }
  }
}
```

#### 前端渲染逻辑
```javascript
for (const [stage, stats] of Object.entries(summary.stages)) {
    // 创建统计行
    // 显示阶段名称和平均耗时
    value.textContent = stats.avg_duration ? stats.avg_duration + 's' : '--';
}
```

### 2.4 🔄 活跃阶段面板

**数据来源:** `summary.global_state` 对象

#### "混音中" (Remixing)
- **数据来源:** `summary.global_state.remixing`
- **后端逻辑:** 统计 `status === "制作中"` 的期数数量

#### "渲染中" (Rendering)
- **数据来源:** `summary.global_state.rendering`
- **后端逻辑:** 统计 `status === "渲染中"` 的期数数量

#### "待制作" (Pending)
- **数据来源:** `summary.global_state.pending`
- **后端逻辑:** 统计 `status === "待制作"` 或 `"已跳过"` 的期数数量

### 2.5 📋 期数列表面板

**API端点:** `GET /metrics/episodes`  
**后端实现:** `dashboard_server.py:get_episodes_status()`

#### 数据流程
```
config/schedule_master.json
    ↓ (StateManager._load())
episodes 数组
    ↓ (状态映射转换)
[
  {
    "episode_id": "20251102",
    "episode_number": 1,
    "schedule_date": "2025-11-02",
    "status": "completed",  // 从 "已完成" 转换
    "title": "Amber Lanterns on Quiet Streets",
    ...
  },
  ...
]
```

#### 显示字段

**期数ID**
- **来源:** `ep.episode_id`（如 `"20251102"`）
- **格式:** `YYYYMMDD`

**标题**
- **来源:** `ep.title`（如 `"Amber Lanterns on Quiet Streets"`）
- **语言:** 英文
- **存储位置:** `schedule_master.json` → `episodes[].title`

**状态**
- **来源:** `ep.status`（已转换为英文：`"completed"`, `"pending"`, `"error"` 等）
- **显示映射:** 
  ```javascript
  const statusDisplay = {
      'pending': '待制作',
      'remixing': '制作中',
      'rendering': '渲染中',
      'uploading': '上传中',
      'completed': '已完成',  // ← 中文显示
      'error': '失败'
  };
  ```
- **视觉标识:** 左侧彩色边框
  - 已完成：绿色 (`border-left-color: #4ade80`)
  - 失败：红色 (`border-left-color: #f87171`)
  - 待制作：灰色 (`border-left-color: #888`)

**操作按钮**
- **条件:** 仅当 `status === "error"` 时显示
- **功能:** "恢复" 按钮
- **API:** `POST /api/recover/{episode_id}`
- **后端逻辑:** `state_manager.rollback_status(episode_id, target_status="pending")`

#### 原始数据结构（schedule_master.json）
```json
{
  "episodes": [
    {
      "episode_id": "20251102",
      "episode_number": 1,
      "schedule_date": "2025-11-02",
      "title": "Amber Lanterns on Quiet Streets",
      "status": "已完成",  // 原始中文状态
      "tracks_used": ["Amber Lanterns...", "Snowfall Serenade", ...],
      "image_path": "/Users/z/Downloads/Kat_Rec/assets/design/images/...",
      "image_used": false,
      "starting_track": "Amber Lanterns Flicker Through Quiet Streets",
      "dominant_color_rgb": [243, 235, 212],
      "metadata_updated_at": "2025-11-02T18:41:39.658615"
    }
  ]
}
```

### 2.6 📰 最近事件面板

**API端点:** `GET /metrics/events?limit=20`  
**后端实现:** `dashboard_server.py:get_events()` 或 `metrics_api.py:get_events()`

#### 数据来源
- **主数据文件:** `data/metrics.json`
- **后端逻辑:** `metrics_manager.get_recent_events(limit=20)`

#### 数据结构（metrics.json）
```json
{
  "events": [
    {
      "timestamp": "2025-11-02T17:41:17",
      "stage": "test",
      "status": "completed",
      "episode_id": "test_20251101",
      "duration": 0.0,
      "metadata": {}
    },
    ...
  ]
}
```

#### 显示字段

**时间戳**
- **来源:** `event.timestamp`
- **格式:** ISO 8601 (`2025-11-02T17:41:17`)
- **前端显示:** `new Date(event.timestamp).toLocaleTimeString()` → `"17:41:17"`

**事件描述**
- **格式:** `{stage} - {status}`
- **示例:** `"test - completed"`, `"unknown - started"`
- **来源组合:**
  - `event.stage`（如 `"test"`, `"remixing"`, `"rendering"`）
  - `event.status`（如 `"completed"`, `"started"`, `"failed"`）

**关联ID**
- **来源:** `event.episode_id` 或 `event.task_id`
- **显示:** 小字号灰色文本

**视觉标识**
- **左侧边框颜色:**
  - 完成：绿色 (`border-left-color: #4ade80`)
  - 失败：红色 (`border-left-color: #f87171`)
  - 其他：蓝色 (`border-left-color: #4a9eff`)

---

### 2.7 底部状态栏

#### "自动刷新: 每10秒 | 最后更新: HH:MM:SS"
- **类型:** 状态提示文本
- **刷新频率配置:**
  ```javascript
  setInterval(refreshDashboard, 10000);  // 10秒 = 10000毫秒
  ```
- **更新时间:** 每次 `refreshDashboard()` 成功后更新

## 3. 歌库管理功能设计

### 3.1 页面目标

**功能页:** `/library/songs` 或 `/library?type=songs`  
**目标用户:** 内容创作者、音频编辑、制作经理  
**核心价值:** 集中管理所有音频素材，追踪使用历史，优化素材选择

### 3.2 数据源分析

#### 主数据文件
- **索引文件:** `data/song_library.csv`
- **使用记录:** `data/song_usage.csv`
- **配置文件:** `config/library_settings.yml`

#### CSV 数据结构（song_library.csv）

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `file_path` | String | 完整文件路径 | `/Users/z/Downloads/LOFI SONGS/305 Sunset Snare.mp3` |
| `file_name` | String | 文件名 | `305 Sunset Snare.mp3` |
| `title` | String | 曲目标题 | `305 Sunset Snare` |
| `duration_seconds` | Float | 时长（秒） | `99.912` |
| `added_at` | ISO DateTime | 添加时间 | `2025-10-15T23:43:52.649286+07:00` |
| `last_used_at` | ISO DateTime | 最后使用时间 | `null` 或 `2025-11-02T10:30:00+08:00` |
| `times_used` | Integer | 使用次数 | `3` |

#### 关联数据（schedule_master.json）
- 每个期数的 `tracks_used` 数组记录了使用的歌曲
- 需要反向查询：给定歌曲标题，查找使用该歌曲的所有期数

### 3.3 核心功能设计

#### 3.3.1 素材列表视图

**布局:** 表格视图（默认）或卡片视图（可选）

**显示列（表格模式）:**

| 列名 | 数据来源 | 说明 | 宽度 |
|------|----------|------|------|
| **选择框** | - | 多选复选框 | 40px |
| **标题** | `title` | 歌曲名称 | 200px |
| **时长** | `duration_seconds` | 格式化显示（mm:ss） | 80px |
| **添加时间** | `added_at` | 日期格式化（YYYY-MM-DD） | 120px |
| **最后使用** | `last_used_at` | 相对时间（"3天前"）或日期 | 120px |
| **使用次数** | `times_used` | 数字 | 80px |
| **状态** | 计算字段 | 可用/已归档/待审核 | 100px |
| **操作** | - | 播放/编辑/删除 | 150px |

**交互行为:**
- **点击标题:** 展开详情侧边栏
- **播放图标:** 内嵌音频播放器预览（需要后端支持音频流）
- **编辑图标:** 打开编辑对话框
- **删除图标:** 确认对话框后删除（软删除，标记为已归档）

**排序功能:**
- 默认：按 `added_at` 倒序（最新添加在前）
- 可排序列：标题、时长、添加时间、最后使用、使用次数
- 排序方式：点击列标题切换升序/降序

**分页:**
- 每页显示：50条（可配置：25/50/100）
- 分页控件：底部显示总页数和页码

---

#### 3.3.2 搜索与筛选

**搜索框:**
- **位置:** 页面顶部
- **功能:** 实时搜索（防抖：300ms）
- **搜索范围:** 
  - `title`（标题）
  - `file_name`（文件名，部分匹配）
- **高亮:** 搜索结果中匹配文本高亮显示

**筛选器组:**
- **位置:** 搜索框下方
- **筛选维度:**
  - **使用状态:** 全部 / 未使用 / 已使用 / 常用（times_used > 5）
  - **时长范围:** 滑块选择（0-600秒）
  - **添加时间:** 日期范围选择器
  - **最后使用:** 全部 / 最近7天 / 最近30天 / 从未使用

**组合筛选:**
- 筛选器可叠加使用
- 显示当前激活的筛选条件标签（可点击删除）

#### 3.3.3 上传功能

**入口按钮:**
- **位置:** 页面右上角，搜索框旁边

## 4. 图库管理功能设计

### 4.1 页面目标

**功能页:** `/library/images` 或 `/library?type=images`  
**目标用户:** 内容创作者、设计师、制作经理  
**核心价值:** 集中管理所有视觉素材，追踪使用历史，优化封面选择

### 4.2 数据源分析

#### 主数据位置
- **物理存储:** `assets/design/images/`（由配置文件指定）
- **使用记录:** `config/schedule_master.json` → `episodes[].image_path`

#### 数据结构（从 schedule_master.json 提取）

```json
{
  "episodes": [
    {
      "image_path": "/Users/z/Downloads/Kat_Rec/assets/design/images/xxx.png",
      "image_used": false,  // 是否已使用
      "dominant_color_rgb": [243, 235, 212],  // 主导色RGB
      "dominant_color_hex": "f3ebd4"  // 主导色HEX
    }
  ]
}
```

**需要扫描的数据:**
- 图片文件列表（通过文件系统扫描）
- 图片元数据（尺寸、文件大小、格式）
- 使用记录（从 schedule_master.json 反向查询）

---

### 4.3 核心功能设计

#### 4.3.1 素材列表视图

## 7. 技术实现要点

### 7.1 后端API设计

#### 7.1.1 歌库API

```
GET    /api/library/songs              # 获取歌曲列表（支持分页、筛选、排序）
GET    /api/library/songs/{id}         # 获取歌曲详情
POST   /api/library/songs/upload       # 上传歌曲
PUT    /api/library/songs/{id}         # 更新歌曲元数据
DELETE /api/library/songs/{id}         # 删除歌曲（软删除）
GET    /api/library/songs/{id}/usage   # 获取歌曲使用记录
```

#### 7.1.2 图库API

```
GET    /api/library/images             # 获取图片列表
GET    /api/library/images/{id}           # 获取图片详情
POST   /api/library/images/upload      # 上传图片
PUT    /api/library/images/{id}         # 更新图片元数据
DELETE /api/library/images/{id}       # 删除图片
GET    /api/library/images/{id}/usage  # 获取图片使用记录
GET    /api/library/images/{id}/preview # 获取图片预览（流式传输）
```

#### 7.1.3 节目库API

```
GET    /api/episodes                   # 获取节目列表
GET    /api/episodes/{id}              # 获取节目详情
POST   /api/episodes                   # 创建新节目
PUT    /api/episodes/{id}              # 更新节目信息
DELETE /api/episodes/{id}              # 删除节目
POST   /api/episodes/{id}/recover      # 恢复失败节目
GET    /api/episodes/{id}/progress     # 获取生产进度
GET    /api/episodes/{id}/events       # 获取事件日志
```

---

### 7.2 前端数据流

```
用户操作
    ↓
前端发起请求 (fetch/axios)
    ↓
后端API处理
    ↓
读取数据文件 (CSV/JSON)
    ↓
数据处理和转换
    ↓
返回JSON响应
    ↓
前端更新DOM
    ↓
用户看到更新
```

---

### 7.3 性能优化建议

**数据加载:**
- **分页加载:** 避免一次性加载大量数据
- **虚拟滚动:** 列表超过1000条时使用虚拟滚动
- **懒加载:** 图片缩略图延迟加载

**缓存策略:**
- **前端缓存:** 使用 localStorage 缓存静态配置
- **后端缓存:** Redis 缓存热点数据（如歌曲列表）

**搜索优化:**
- **防抖:** 搜索输入防抖300ms
- **后端搜索:** 复杂搜索在后端执行，避免前端遍历
